# Queries

> [!NOTE] Any expressions listed below are only supported for **top level properties and list items**.

As you might have seen in the [Clients section](Clients.md), some methods can make use of special **filter expressions** which allow you to use **database functions** or special Cypher syntax in a easy way.


## Property paths

Paths in queries can easily be accessed via their property path. This can be achieved by simply accessing the property in the model class directly:

```python
descriptor = Human.metadata.jobs.name
```

For more information on how property paths work for lists, see [Collections expressions](#collections-expressions).


## Filter expressions

> [!NOTE] You may notice that certain database functions are not natively implemented by loomi. This is due to some functions only being available for either Neo4j or Memgraph. Check the [custom expressions sections](#writing-your-own-expressions) to see how you can implement your own database functions to be used with loomi query builders.

Loomi provides a number of filter expressions which can be used with different types of query builders (see the [Clients section](./Clients.md#fetching-entities)). All expressions which are supported can be divided into the following groups:
- **Arithmetic**: Adding, subtracting, etc.
- **Collection**: Used when filtering on nested paths (mostly relevant for Memgraph)
- **Comparison**: Equals, logical operators, string comparisons, etc.
- **Identity**: ID and element ID related operators
- **Transformation**: Rounding, string trimming, etc.

> [!NOTE] The following will show how to use operator overloading. This may cause incorrect type errors. You will not run into these errors when using the functions instead.

> [!NOTE] All Cypher example in the following will show the value directly in the query. The actual queries generated by the templating engine will always use parameters instead.

### Arithmetic expressions

| Function                           | Overloaded Operator | Cypher      |
| :--------------------------------- | :------------------ | :---------- |
| `add(Human.age, 2)`                | `Human.age + 2`     | `e.age + 2` |
| `reflected_add(Human.age, 2)`      | `2 + Human.age`     | `2 + e.age` |
| `subtract(Human.age, 2)`           | `Human.age - 2`     | `e.age - 2` |
| `reflected_subtract(Human.age, 2)` | `2 - Human.age`     | `2 - e.age` |
| `multiply(Human.age, 2)`           | `Human.age * 2`     | `e.age * 2` |
| `reflected_multiply(Human.age, 2)` | `2 * Human.age`     | `2 * e.age` |
| `divide(Human.age, 2)`             | `Human.age / 2`     | `e.age / 2` |
| `reflected_divide(Human.age, 2)`   | `2 / Human.age`     | `2 / e.age` |
| `modulo(Human.age, 2)`             | `Human.age % 2`     | `e.age % 2` |
| `reflected_modulo(Human.age, 2)`   | `2 % Human.age`     | `2 % e.age` |
| `pow_(Human.age, 2)`               | `Human.age ^ 2`     | `e.age ^ 2` |
| `reflected_pow(Human.age, 2)`      | `2 ^ Human.age`     | `2 ^ e.age` |

These methods can be called with their respective functions:

```python
expr = add(Human.age, 2)
```

or by using their operator overload:

```python
expr = Human.age + 2
```


### Collections expressions

These expressions are mainly used for filtering on list items and are only used to determine which operator to use when list items are filtered on.

| Function             | Overloaded Operator | Cypher                           |
| :------------------- | :------------------ | :------------------------------- |
| `all_(Human.jobs)`   | -                   | `ALL(v0 IN e.jobs WHERE ...)`    |
| `any_(Human.jobs)`   | -                   | `ANY(v0 IN e.jobs WHERE ...)`    |
| `none(Human.jobs)`   | -                   | `NONE(v0 IN e.jobs WHERE ...)`   |
| `single(Human.jobs)` | -                   | `SINGLE(v0 IN e.jobs WHERE ...)` |

By default, if you access a path like `Human.jobs.name` (`Human.jobs` being a list here), the `any_` function will be used. You can choose to use any other operator with their respective functions:

```python
expr = equals(none(Human.jobs).name, "Developer")
```


#### Index based access

You can also filter on a specific list item if you know it's index. This does not have a specific function, but is rather written like you would access the index normally:

```python
expr = equals(Human.jobs[2].name, "Developer")
```


### Comparison expressions

The expressions allow you to use a number of comparison operators supported by both Neo4j and Memgraph.

| Function                                | Overloaded Operator                       | Cypher                                  |
| :-------------------------------------- | :---------------------------------------- | :-------------------------------------- |
| `equals(Human.name, "John")`            | `Human.name == "John"`                    | `e.name = 'John'`                       |
| `not_equals(Human.name, "John")`        | `Human.name != "John"`                    | `e.name <> 'John'`                      |
| `greater_than(Human.age, 24)`           | `Human.age > 24`                          | `e.age > 24`                            |
| `greater_than_or_equals(Human.age, 24)` | `Human.age >= 24`                         | `e.age >= 24`                           |
| `less_than(Human.age, 24)`              | `Human.age < 24`                          | `e.age < 24`                            |
| `less_than_or_equals(Human.age, 24)`    | `Human.age <= 24`                         | `e.age <= 24`                           |
| `not_(<sub-expression>)`                | `~(<sub-expression>)`                     | `NOT(<sub-expression>)`                 |
| `is_null(Human.name)`                   | -                                         | `e.name IS NULL`                        |
| `is_not_null(Human.name)`               | -                                         | `e.name IS NOT NULL`                    |
| `in_(Human.name, ["Jane", "John"])`     | -                                         | `e.name IN ['Jane', 'John']`            |
| `starts_with(Human.name, "Jo")`         | -                                         | `e.name STARTS WITH 'Jo'`               |
| `ends_with(Human.name, "hn")`           | -                                         | `e.name ENDS WITH 'hn'`                 |
| `regex(Human.name, ".*hn")`             | -                                         | `e.name =~ '.*hn'`                      |
| `and_(<sub-expression>, ...)`           | `(<sub-expression>) & (<sub-expression>)` | `<sub-expression> AND <sub-expression>` |
| `or_(<sub-expression>, ...)`            | `(<sub-expression>)                       | (<sub-expression>)`                     | `<sub-expression> OR <sub-expression>` |
| `xor(<sub-expression>, ...)`            | `(<sub-expression>) ^ (<sub-expression>)` | `<sub-expression> XOR <sub-expression>` |



### Identity expressions

These expressions allow you to reference the database IDs of models in queries.

> [!NOTE] When using Memgraph, the `id_` function will always be used, as Memgraph does not support the `elementId` database function.

| Function            | Overloaded Operator | Cypher         |
| :------------------ | :------------------ | :------------- |
| `element_id(Human)` | -                   | `elementId(e)` |
| `id_(Human)`        | -                   | `id(e)`        |


### Transformation expressions

These expressions allow you to transform the value with a database function.

| Function               | Overloaded Operator | Cypher            |
| :--------------------- | :------------------ | :---------------- |
| `tail(Human.name)`     | -                   | `tail(e.name)`    |
| `abs_(Human.age)`      | -                   | `abs(e.age)`      |
| `ceil(Human.age)`      | -                   | `ceil(e.age)`     |
| `floor(Human.age)`     | -                   | `floor(e.age)`    |
| `round_(Human.age)`    | -                   | `round(e.age)`    |
| `ltrim(Human.name)`    | -                   | `ltrim(e.name)`   |
| `rtrim(Human.name)`    | -                   | `rtrim(e.name)`   |
| `trim(Human.name)`     | -                   | `trim(e.name)`    |
| `to_lower(Human.name)` | -                   | `toLower(e.name)` |
| `to_upper(Human.name)` | -                   | `toUpper(e.name)` |


### Writing your own expressions

You might have some complex queries which would be hard to reflect using the build-in expressions. In that case you can write your own expressions which can then be compiled by the client like any other. For this, you can use the `.cypher()` method:

```python
# This method recreates the build-in `in_` expression
custom_expr = cypher(
  "{human}.name IN {list_values}",
  {"human": Human},
  {"list_values": ["Monty", "John", "James"]},
)
```

As you can see, your custom cypher method must define at least one part, and optionally 2 more:
- **Template**: This is a template for your custom Cypher expression. Any variables or parameters can be defined with placeholder values. In the example above, `human` defines a variable and `list_values` defines a parameter.
- **Variable references**: This is where you define which placeholder values should be filled with the corresponding variable name of which model. This essentially tells the query template engine what models this variable should map to.
- **Parameter references**: You could define any values you need directly in the template, but a better approach is to use parameter placeholders. Any placeholder name defined here will be replaced with a parameter when the template is compiled.

These expressions can contain any query which is valid after the `WHERE` keyword.


#### Aliases

Since this method is flexible, you might run into a situation where you need to reference the same model but with a different variable name. This could be the case for query which use the `EXISTS` keyword or subqueries. For this you can create aliases for models:

> [!NOTE] Aliased models can not have a name in the format `v<number>`, as this format is reserved for internal use.

```python
# Creates a alias
aliased_human = create_alias(Human, "my_alias")

# This is equal to a Cypher expression like `my_alias.name IN $p<generated-number>`
custom_expr = cypher(
  "{human}.name IN {list_values}",
  {"human": aliased_human},
  {"list_values": ["Monty", "John", "James"]},
)
```

Aliased models will use their assigned name as variable names in queries, so you can even reference them yourself if you need to.
